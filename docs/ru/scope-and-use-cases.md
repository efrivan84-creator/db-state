# Область применения и сценарии использования

> [English](../en/scope-and-use-cases.md) · **Русский**

Эта страница отвечает на вопрос, который возникает до того как кто-то попробует библиотеку: **«Подходит ли db-state для *моего* проекта?»**

Если коротко: **db-state покрывает 70-90% realtime-состояния в почти любом проекте — даже в тех, где нужны ultra-low-latency каналы (игры, голос, live cursors)**. Библиотека спроектирована как **базовый фундаментальный слой**; скорость-критичные фичи дописываются тонким custom-слоем поверх **того же сокета**.

Эта страница объясняет модель «слоёв», показывает где db-state помещается в разных типах проектов, и честно говорит где проходит граница.

## Модель слоёв

Почти любой нетривиальный realtime-продукт имеет состояние с **разными требованиями к latency**:

```
┌─────────────────────────────────────┐
│ Ultra-low-latency слой              │  ← custom-код, ~50-500 строк на проект
│  - позиции, попадания, голос, курсоры│
├─────────────────────────────────────┤
│ State / sync слой (db-state)        │  ← @db-state/vue: 5.4 KB brotli
│  - профили, инвентарь, очки         │
│  - чат, matchmaking, лобби          │
│  - настройки, achievements, история │
├─────────────────────────────────────┤
│ Persistence (MongoDB)               │
└─────────────────────────────────────┘
```

db-state обрабатывает средний слой — персистентное, авторитативное, с проверкой прав, реактивное состояние, которое нужно синхронизировать между множеством клиентов, но не нужно с задержкой <50ms.

Поверх него ты пишешь тонкий custom-протокол — **через тот же WebSocket**, который библиотека уже использует — для всего, что должно быть моментальным.

## Почему это работает именно с db-state

Большинство realtime-библиотек не дают такого слоения легко:

| Альтернатива | Почему layering custom поверх — сложно |
|---|---|
| Firebase / Firestore | Закрытый протокол; для своего слоя — отдельный WebSocket с отдельной аутентификацией. |
| Convex / InstantDB | Managed-сервис, нет голого сокета. |
| Hasura / Supabase | GraphQL/REST API; realtime — отдельный Phoenix-style канал. |
| RxDB + CouchDB | Нет каналов событий — только репликация документов. |
| Apollo + GraphQL Subscriptions | Подписки идут через Apollo; для своего протокола — параллельный WebSocket. |

В db-state это делается тривиально:

```js
// Состояние (через db-state)
const player = state.player.load(userId)
await state.player.update({ id: userId, set: { inventory: [...] } })

// Custom ultra-fast канал (тот же сокет, твой протокол)
state.socket.on("position:update", (msg) => {
  applyOpponentPosition(msg.payload)
})

state.socket.send("position:update", {
  x, y, z, ts: performance.now()
})
```

Один сокет. Одна авторизация. Два протокола на двух скоростях.

Префикс `dbstate:*` зарезервирован за библиотекой; всё остальное — твоё. Auth проверяется один раз; оба слоя видят один `userId`. Никакого второго соединения, никакого второго handshake, никаких CORS-проблем.

## Что db-state покрывает в разных типах проектов

### Админки / CRM / dashboards

**Custom-кода поверх db-state: 0 строк.**

Библиотека делает всё: типизированные реактивные таблицы, CRUD, логин, sync, права, офлайн-кэш. Ты пишешь `state.order.update({...})` — и UI обновляется везде.

Это канонический use case (см. [demo2](../../demo2)).

### Collaborative редакторы (Notion / Linear / kanban-стиль)

**Custom-кода: ~50 строк для позиций курсоров и выделений.**

- Документы, блоки, проекты, задачи → db-state
- Комментарии, упоминания, история → db-state
- Sharing-permissions → `_permission` таблица db-state
- Позиции курсоров, «X печатает», подсветки выделений → custom-канал через `state.socket`

Custom-слой тут маленький, потому что курсоры эфемерны — нет смысла писать каждое движение курсора в Mongo.

### B2B приложения (CRM, ERP, мониторинг-дашборды)

**Custom-кода: 0 строк** для большинства случаев. **~50-100 строк** если есть live-метрики, обновляющиеся чаще раза в секунду.

Стандартные таблицы (сделки, клиенты, тикеты, алерты) синхронизируются через db-state. Live KPI-тикеры могут обновляться через `state.metric.update(...)` с уменьшенным debounce на этой таблице, или через custom-канал событий.

### Browser multiplayer (карточные игры, party-games, стратегии)

**Custom-кода: ~100-200 строк для синхронизации ходов / действий.**

- Метаданные матча, игроки, очки, чат → db-state
- Изменения состояния игры при действиях игроков → db-state
- Синхронизация ходов, optimistic moves → custom-канал для низкой задержки
- Финальное разрешение хода → db-state мутация (authoritative)

20Hz position-updates для 10 игроков работают комфортно при `changesBroadcastDelay: 50ms` на соответствующей таблице.

### Real-time аукционы / bidding / live-commerce

**Custom-кода: ~100-200 строк для дифф'ов order-book / bid-tape.**

- Листинги, товары, балансы пользователей, история транзакций → db-state
- Live price-тикеры и обновления ставок → db-state с низким debounce, или custom-канал
- «X сделал ставку» уведомления → app-level event через `state.socket`

### Trading платформы

**Custom-кода: ~200-500 строк для стриминга order book и chart-данных.**

- Состояние счёта, позиции, ордера, настройки, история, watchlist'ы → db-state
- Order book level diffs с частотой 10-100ms → custom-канал для скорости
- Chart-свечи → кэшируются через db-state, обновления — через custom-канал

### MMO / casual real-time игры

**Custom-кода: ~500-1000 строк для протокола движения и боя.**

- Профили игроков, инвентарь, гильдии, achievements, друзья, leaderboards → db-state
- Результаты матчей, post-game статистика, XP, валюта → db-state
- Real-time позиции, действия, попадания → custom бинарный (или почти бинарный) протокол через тот же сокет
- Лобби, matchmaking, чат → db-state

### FPS / hardcore action игры

**Custom-кода: ~2000+ строк для полноценного сетевого стэка.**

Здесь custom-слой становится существенным — нужны client-side prediction, lag compensation, anti-cheat hooks, server-authoritative физика. Это всё не задача db-state.

Но **метагейм** — всё, что вне игрового loop'а — всё ещё естественная территория db-state:

- Аккаунт, друзья, party, matchmaking → db-state
- Loadout, скины, валюта, battle pass → db-state
- История матчей, метаданные replays, статистика → db-state
- Магазин, leaderboards, achievements → db-state
- Чат в лобби, post-game чат → db-state

В типичном FPS-кодбейзе метагейм — это 60-80% всего backend-стейта. db-state забирает это бесплатно; ты фокусируешься на 20%, которые реально специфичны для игры.

## Где db-state не подходит

Есть чёткая граница. Некоторые категории не подходят, и их не нужно сюда тянуть:

- **CRDT-based коллаборативное редактирование текста** (Google Docs, Figma-like канвас). Используй [Yjs](https://yjs.dev) или [Automerge](https://automerge.org) — они решают другую задачу (multi-master merge с разрешением конфликтов на уровне символов).
- **Voice / video streaming**. WebRTC, а не WebSocket.
- **High-frequency time-series ingest**. Специализированные хранилища типа ClickHouse / TimescaleDB; db-state не построен для миллионов write/sec.
- **React / Svelte / Solid фронтенды**. У db-state только Vue 3 клиент. Ядро (core) framework-agnostic — порт возможен — но сегодня его нет в коробке.
- **PostgreSQL / SQLite backends**. db-state поставляет только `@db-state/server-mongo`. Протокол можно реализовать поверх Postgres (через logical replication или триггеры), но опять же — сегодня этого нет.
- **Authoritative игровая логика** (физика, anti-cheat, lag compensation). Библиотека — это state-sync транспорт, не игровой движок.

Если **весь твой продукт** — из этих категорий, db-state — неправильный инструмент. Если только **часть** — db-state всё равно покроет всё остальное.

## Практическая таблица покрытия

Сколько realtime-состояния твоего проекта в строках кода берёт на себя db-state?

| Тип проекта | db-state покрывает | Custom-код поверх | Почему |
|---|---:|---:|---|
| Админка / CRM / dashboard | 100% | 0 строк | Чистый CRUD с правами |
| Collaborative редактор | ~95% | ~50 строк | Только позиции курсоров и live-подсветки |
| Browser multiplayer (casual) | ~85% | ~100-200 строк | Sync ходов / действий — нужна низкая задержка |
| Real-time аукцион | ~90% | ~100-200 строк | Стриминг bid-tape |
| Trading платформа | ~80% | ~200-500 строк | Order book + графики |
| Casual MMO | ~70% | ~500-1000 строк | Протокол движения |
| FPS / action игра | ~60% | ~2000+ строк | Полный игровой сетевой стэк |

В каждой строке db-state берёт **большинство** realtime-состояния. Именно поэтому архитектура «db-state как фундамент + тонкий custom-слой для скорости» работает даже для проектов, которые на первый взгляд не имеют ничего общего.

## Почему один сокет это важно

Самое недооценённое свойство этой модели — что **всё идёт через один WebSocket**:

- Одно TCP-соединение в обслуживании.
- Одна стратегия реконнекта.
- Одна аутентификация — оба слоя видят один `userId`.
- Один latency-бюджет на сам сокет.
- Одна точка наблюдаемости.
- Одно firewall-правило.
- Одна точка TLS-терминирования.

Сравни с проектами, где realtime разделён между «Firestore для стейта + Socket.IO для чата + WebRTC для голоса + custom для игры» — каждая часть со своей аутентификацией, со своим реконнектом, со своим мониторингом. Mental load растёт линейно с числом каналов.

С db-state твой custom-слой — это несколько вызовов `state.socket.on(...)` и `state.socket.send(...)` в том же коде, который уже общается с базой.

## Итог

Почти любой продукт можно построить с db-state как фундаментом. Библиотека честна в том, что не пытается быть игровым движком, CRDT-решателем или голосовым транспортом — для этого есть правильные инструменты. Но для **всего, что попадает в шаблон «персистентное состояние, синхронизируемое между множеством клиентов, с правами и офлайн-кэшем»** — db-state покрывает.

Для проектов, которым нужны ещё и ultra-low-latency каналы, они **слоятся сверху чисто** — никакого второго соединения, никакой второй аутентификации, никакого взрыва протоколов. Ты пишешь, скажем, 50-500 строк на скорость-критичную часть и отдаёшь db-state другие 70-90% состояния твоего проекта бесплатно.

Это делает db-state **базовым слоем**, а не нишевым инструментом. Ниша — это предположение.
